# 运行&开发约束规则
## 一、环境与网络约束
1. 服务端强制兼容 Windows 10 / Windows 11，所有代码、配置、打包产物禁止使用Linux/Mac专属语法。
2. 服务默认监听 `0.0.0.0`，必须支持**局域网内网跨设备访问**，核心能力不可删减。
3. 定位为内网工具，**禁止开发公网部署、端口映射、外网穿透**相关功能。
4. 运行要求：低内存占用，普通Windows主机即可稳定运行，支持多终端同时访问。
5. 端口：默认8080，支持自定义修改，必须提供端口修改配置说明。

## 二、功能约束
1. 基础功能强制保留：词条新增、本地持久化存储、多字段智能检索、WebUI交互、一键复制。
2. 检索逻辑：同时匹配标题+内容，支持精准匹配、模糊分词检索。
3. 前端必须做**自适应适配**，保证电脑、手机、平板等局域网终端正常显示与操作。
4. 数据安全：所有数据仅存储在Windows本地，不上传至任何网络服务器。
5. 精简原则：非需求功能不额外新增特效、广告、会员、云同步等冗余模块。

## 三、代码与文件约束
1. 目录结构清晰，关键代码添加中文注释，方便二次修改与维护。
2. 全局统一 UTF-8 编码，彻底解决Windows环境中文乱码问题。
3. 所有文件路径使用**相对路径**，程序可在Windows任意目录移动、运行。
4. 第三方依赖尽量精简，降低环境配置难度，依赖清单统一写入requirements.txt。

## 四、部署与输出约束
1. 必须配套完整**局域网部署教程**：包含Windows查看内网IP、防火墙端口放行、访问地址格式说明。
2. 代码、脚本、程序包保证完整可直接运行，部署步骤通俗易懂，零基础可操作。
3. 网络类问题必须提供排错方案：端口占用、防火墙拦截、IP错误、跨设备无法访问等。
4. 迭代修改时，优先保证原有局域网访问能力正常，再新增功能。

## 五、交互约束（面向OpenClaw）
1. 专注当前工具的开发、部署、运维、迭代，不回应无关问题。
2. 需求模糊时，仅询问功能、界面、端口、网络配置四类核心信息。
3. 严格遵循TECH_STACK、WORKFLOW规范，不擅自更换技术栈与整体架构。
4. 严格依照用户提供的参考资料进行迭代，不脱离资料自主新增功能。
